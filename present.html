SOP-01: ฝั่งลูกค้า (Customer Flow)

Customer

WI-C01 — สมัครสมาชิก (Register)

ลูกค้าใหม่ที่ยังไม่มีบัญชี กรอกข้อมูลผ่านฟอร์ม Register บน index.html

ข้อมูลที่ต้องกรอก (Input)
  • ชื่อ-นามสกุลจริง (full_name)
  • ชื่อผู้ใช้งาน (username) — ห้ามซ้ำ
  • รหัสผ่าน (password)
  • เบอร์โทรศัพท์ (phone) — ใช้กู้คืนรหัสผ่าน
ผลลัพธ์ (Output)
  • User ID: U-XXXXXXXX (8 หลัก)
  • Role: customer (กำหนดอัตโนมัติ)
  • Session เก็บลง localStorage
  • Email แจ้งเจ้าของร้านมีสมาชิกใหม่
Backend: POST action=register → ตรวจ username ซ้ำ → append USERS sheet → sendNotify()
Validation: ระบบตรวจสอบ username ซ้ำก่อน insert — ถ้าซ้ำจะ return error "ชื่อผู้ใช้งานนี้ถูกใช้ไปแล้ว" โดยไม่บันทึก

WI-C02 — เข้าสู่ระบบ (Login)

ตรวจสอบ username + password กับ USERS sheet โดยตรง

การล็อกอิน
  • กรอก username + password
  • ติ๊ก "จดจำฉัน" → บันทึกลง localStorage
  • ไม่ติ๊ก → บันทึกลง sessionStorage (ปิดแล้วหาย)
ข้อมูลที่ได้รับหลัง Login
  • user_id, username
  • role: customer / tailor / boss
  • full_name
  • Demo: admin / 1234 (bypass hardcoded)
Backend: POST action=login → เปรียบเทียบ USERS sheet → คืน user object (ไม่ส่ง password กลับ)
หากล็อกอินด้วย role=boss → ระบบส่ง Email แจ้งเตือน "มีการเข้าสู่ระบบ BOSS"

WI-C03 — สั่งงานจ้าง (Place Order)

ลูกค้ากรอกแบบฟอร์มสั่งงาน — ระบบสร้างใบงานและแจ้งเจ้าของร้านทันที

ข้อมูลในใบสั่งงาน
  • ชื่อลูกค้า + เบอร์โทร + Line ID
  • รายละเอียดงาน (≥ 10 ตัวอักษร)
  • ประเภทชุด (เดรส / สูท / กางเกง ฯลฯ)
  • งบประมาณ (ตัวเลข > 0)
  • สัดส่วนร่างกาย: อก / เอว / สะโพก / ยาว / ไหล่
ผลลัพธ์
  • Job ID: UUID (xxxxxxxx-xxxx-...)
  • เลขที่ใบงาน: RN-XXXX-XXXX
  • สถานะเริ่มต้น: pending
  • Modal ใบเสร็จดิจิทัล (copy RN)
  • Email แจ้งร้าน: ลูกค้า / งาน / งบประมาณ
Backend: POST action=create → uuid() + RN-XXXX-XXXX → appendRow JOBS → sendNotify()

WI-C04 — ติดตามสถานะ (Job Tracking)

ลูกค้าดูงานของตัวเองได้ทันที ระบบกรอง user_id อัตโนมัติ

สีแสดงสถานะ
  • เหลือง = รอรับงาน
  • ฟ้า = รับงานแล้ว
  • ส้ม = กำลังเย็บ
  • เขียว = เสร็จสมบูรณ์
  • แดง = ยกเลิก/ปฏิเสธ
ข้อมูลที่เห็นในใบงาน
  • เลขที่ RN + สถานะ
  • ชื่อช่างที่รับงาน
  • รูปความคืบหน้า (URL)
  • วันที่สร้าง
  • เหตุผล (กรณียกเลิก)
ปุ่มที่ใช้ได้ตามสถานะ
  • pending → ยกเลิกงานได้
  • accepted → ยกเลิกได้ + ดูช่าง
  • sewing → ดูรูปความคืบหน้า
  • finished → ยืนยัน + Rating
Frontend: ดึงข้อมูลจาก GET (doGet) → filter jobs ที่ user_id ตรงกัน

WI-C05 — ยืนยันรับงาน + ให้คะแนน (Confirm & Rate)

เมื่องานสถานะ finished ลูกค้ากดยืนยันรับงานและให้ Rating

ขั้นตอน
  • กดปุ่ม "ยืนยันรับงาน" บนใบงาน
  • เลือกดาว 1-5 (5 = ดีมาก)
  • กด confirm → ส่งข้อมูล
ผลกระทบ
  • คอลัมน์ rating ถูกบันทึกลง JOBS
  • คะแนนปรากฏในสถิติผลงานช่าง
  • ช่างเห็นคะแนนในแท็บ "รายได้ของฉัน"
Backend: POST action=update_status, status=finished, rating=X → บันทึก col 14 ของ JOBS

WI-C06 — ยกเลิกงาน (Cancel Order)

ยกเลิกได้เฉพาะงานสถานะ pending หรือ accepted

เงื่อนไข
  • สถานะต้องเป็น pending หรือ accepted
  • ต้องระบุเหตุผลการยกเลิก
  • หลังยกเลิก: ไม่สามารถยกเลิกคำสั่งยกเลิกได้
ผลกระทบ
  • Status → cancelled
  • เหตุผลบันทึก col cancellation_reason
  • Email แจ้งเจ้าของร้านทันที
  • สต๊อกที่หักไปแล้ว ไม่ถูกคืน (Boss ต้องแก้เอง)
Backend: POST action=update_status, status=cancelled, cancellation_reason="..." → sendNotify()

WI-C07 — ส่งข้อร้องเรียน/เคลมงาน (Dispute)

ลูกค้าพบปัญหางานไม่ตรงตามสั่ง สามารถส่ง note ถึงช่างและเจ้าของร้าน

วิธีใช้
  • กดปุ่ม "แจ้งปัญหา/เคลม" บนใบงาน
  • กรอกรายละเอียดปัญหา
  • ส่ง dispute_notes
ผลลัพธ์
  • บันทึกลง col dispute_notes ใน JOBS
  • Boss เห็นใน Tab วิเคราะห์ระบบ
  • ไม่เปลี่ยน status อัตโนมัติ
Backend: POST action=update_status, dispute_notes="..." → update col 18 ของ JOBS

WI-C08 — ลืมรหัสผ่าน (Forgot Password)

ยืนยันตัวตนด้วย username + เบอร์โทรที่ลงทะเบียนไว้ แล้วตั้งรหัสใหม่

ขั้นตอน
  • กดลิงก์ "ลืมรหัสผ่าน?" ที่หน้า Login
  • กรอก username + เบอร์โทร + รหัสใหม่
  • ระบบตรวจสอบ username และ phone ตรงกันใน USERS
ผลลัพธ์
  • อัปเดต password ใหม่ใน USERS sheet
  • บันทึก Log: RESET_PASSWORD
  • Email แจ้ง "กู้คืนรหัสผ่านสำเร็จ"
Backend: POST action=reset_password → verify username+phone → update col 3 ของ USERS → logEvent()

SOP-02: ฝั่งช่างเย็บผ้า (Tailor Flow)

Tailor

WI-T01 — เข้าสู่ระบบและดูงาน (Login & View Jobs)

ช่างเข้าระบบผ่าน tailor.html เท่านั้น — ต่างจาก index.html ที่เป็นฝั่งลูกค้า

หน้าที่เห็นหลัง Login (Role: tailor)
  • Tab "งานใหม่" — งาน pending รอรับ
  • Tab "กำลังเย็บ" — งานที่รับแล้ว (accepted/sewing)
  • Tab "รายได้ของฉัน" — สรุปรายได้ + เบิกเงิน
  • Badge แจ้งเตือนจำนวนงานใหม่
หน้าที่ไม่เห็น (Boss only)
  • Tab ภาพรวม / สต๊อก / ผลงานช่าง
  • Tab การเงิน / P&L / วิเคราะห์
  • การจัดการสิทธิ์ผู้ใช้
  • Boss Navigation Hub ด้านบน
Frontend: ตรวจสอบ role จาก localStorage → ซ่อน/แสดง element ที่มี class .boss-only

WI-T02 — รับงาน + หักสต๊อก (Accept Job & Deduct Stock)

ช่างพิจารณางานใหม่ กดรับงาน พร้อมเลือกวัสดุจากคลังสต๊อก

ขั้นตอนการรับงาน
  • ดูรายละเอียดงาน: ประเภท / สัดส่วน / งบ
  • กดปุ่ม "รับงาน"
  • เลือก Stock Item + ระบุจำนวนที่ใช้
  • กดยืนยัน
สิ่งที่ระบบทำอัตโนมัติ
  • Status → accepted
  • บันทึก tailor_name ลง JOBS
  • หักจำนวนสต๊อก (qty - qtyUsed)
  • คำนวณ material_cost = qty × unit_price
  • ถ้า qty ≤ min_threshold → Email เตือน "ของใกล้หมด"
Backend: POST action=update_status, status=accepted, stock_id, stock_qty → update STOCK (col4,8) + update JOBS (col15: material_cost)
สต๊อกจะถูกหักเฉพาะเมื่อ stock_id ถูกส่งมาพร้อมกับ status=accepted เท่านั้น — ถ้าไม่เลือกวัสดุ material_cost จะเป็น 0

WI-T03 — ปฏิเสธงาน (Reject Job)

ช่างไม่สามารถรับงานได้ ต้องระบุเหตุผลทุกครั้ง

เหตุผลที่พบบ่อย
  • วัสดุที่ลูกค้าระบุไม่มีในสต๊อก
  • งานเกินความสามารถหรือไม่ถนัด
  • งานซ้ำซ้อนกับที่รับไปแล้ว
  • งบประมาณต่ำเกินไป
ผลกระทบ
  • Status → rejected
  • บันทึกเหตุผลลง cancellation_reason
  • Email แจ้งเจ้าของร้าน
  • ปรากฏในแท็บ "วิเคราะห์ปัญหา" ของ Boss
Backend: POST action=update_status, status=rejected, cancellation_reason="..." → sendNotify() + logEvent()

WI-T04 — อัปเดตความคืบหน้า (Progress Update)

ช่างอัปโหลดรูปภาพงาน และเปลี่ยนสถานะเป็น "กำลังเย็บ"

การอัปโหลดรูปภาพ
  • อัปโหลดรูปไป Google Drive ก่อน
  • คัดลอก Shareable Link มาวาง
  • ระบบบันทึก URL ลง col progress_photo
  • ลูกค้าเห็นรูปแบบ real-time (หลัง refresh)
ผลลัพธ์
  • Status → sewing
  • รูปปรากฏในใบงานของลูกค้า
  • บันทึก Log: UPDATE_STATUS → sewing
Backend: POST action=update_status, status=sewing, progress_photo="URL" → update col9 + col17 ของ JOBS

WI-T05 — แจ้งงานเสร็จ (Mark Finished)

ช่างทำงานเสร็จสมบูรณ์ กดแจ้งเสร็จ — ระบบ notify เจ้าของร้านเพื่อนัดส่งมอบ

เงื่อนไข
  • สถานะต้องเป็น accepted หรือ sewing
  • ไม่จำเป็นต้องมีรูปความคืบหน้า
ผลกระทบ
  • Status → finished
  • Email: "งานเสร็จแล้ว! พร้อมส่งมอบ"
  • ลูกค้าเห็นปุ่ม "ยืนยันรับงาน + Rating"
  • รายได้ช่างเพิ่มขึ้นใน Tab "รายได้ของฉัน"
Backend: POST action=update_status, status=finished → sendNotify("✅ งานเสร็จแล้ว...") + logEvent()

WI-T06 — เบิกเงิน (Request Payroll)

ช่างยื่นคำขอเบิกเงินค่าแรงในรอบที่กำหนด รอ Boss อนุมัติ

การคำนวณรายได้ช่าง
  • รายได้รวม = Σ (budget × 40%) ของงาน finished
  • เลือกรอบ: วันที่ 15 หรือ 30 ของเดือน
  • Gross = ยอดรวมที่ขอเบิก
  • Tax 3% = ภาษีหัก ณ ที่จ่าย
  • Net = Gross - Tax
สถานะคำขอ
  • pending = รอ Boss อนุมัติ
  • approved = Boss อนุมัติแล้ว
  • Pay ID: PAY-XXXXXXXX
  • บันทึกทุกรายการใน PAYROLL sheet
Backend: POST action=request_payment → คำนวณ gross/tax/net → appendRow PAYROLL (status=pending)
หมายเหตุ: ระบบไม่ล็อคงานหลังเบิก — Boss ควรตรวจสอบว่าช่างไม่เบิกซ้ำรอบเดิม

SOP-03: ฝั่งเจ้าของร้าน (Boss Flow)

Boss

WI-B01 — ตรวจ Dashboard รายวัน

เปิดแอปทุกเช้า ตรวจสอบสถานะโดยรวม ก่อนเริ่มงาน

การ์ดแถวบน (วันนี้)
  • งานวันนี้ (สร้างวันนี้)
  • รายได้วันนี้ (budget รวม)
  • งานเสร็จวันนี้
สถิติสะสม
  • รายได้รวมทั้งหมด
  • งานค้างในระบบ (pending)
  • กำลังตัดเย็บ (accepted+sewing)
  • ทำสำเร็จทั้งหมด
  • จำนวนปฏิเสธ
กราฟ (Lazy Load)
  • กราฟรายได้รายเดือน (Bar)
  • สัดส่วนประเภทชุด (Pie/Doughnut)
  • มูลค่าสต๊อกแยกหมวด (Bar)
  • ค่าแรงช่าง 40/60 ต่อคน
Frontend: คำนวณทั้งหมดจาก jobs/stock array ที่ดึงมาจาก doGet — ไม่มี API แยกสำหรับ stats

WI-B02 — จัดการสต๊อกวัสดุ (Stock Management)

การดูและกรองสต๊อก
  • ดูรายการทั้งหมดใน Tab "สต๊อก"
  • ติ๊ก "กรองเฉพาะของใกล้หมด" → แสดงเฉพาะที่ qty ≤ min_threshold
  • แก้ไขจำนวนได้ inline โดยคลิกที่ตัวเลข
เพิ่มรายการสต๊อกใหม่
  • กดปุ่ม "+ เพิ่มรายการ"
  • กรอก: ชื่อ, หมวดหมู่, จำนวน, หน่วย, ราคา/หน่วย, ขั้นต่ำ
  • Item ID: STK-XXXXXX (สร้างอัตโนมัติ)
Backend: POST action=add_stock → STK-UUID → appendRow STOCK | POST action=update_stock → update col4+col8
ข้อควรระวัง: ระบบไม่มีการคืนสต๊อกอัตโนมัติเมื่อยกเลิกงาน Boss ต้องปรับจำนวนเองหากมีการยกเลิกหลังรับงาน

WI-B03 — ดูผลงานและค่าแรงช่าง (Performance)

วิเคราะห์ผลงานแต่ละคนในแท็บ "ผลงานช่าง" — ใช้สำหรับประเมินและคิดค่าแรง

ข้อมูลที่แสดงต่อช่าง 1 คน
  • จำนวนงานที่ทำเสร็จ
  • ยอดรายได้รวม (budget sum)
  • ค่าแรง 40% = budget × 0.4
  • ต้นทุนวัสดุที่หักไป (material_cost)
  • กำไรร้านสุทธิ = (budget × 0.6) - material_cost
  • Rating เฉลี่ย
การใช้งาน
  • ใช้ประกอบการพิจารณาเบิกเงิน
  • ตรวจสอบประสิทธิภาพช่างแต่ละคน
  • วางแผนรับงานตามความสามารถ
Frontend: group jobs by tailor_name ที่ status=finished → คำนวณทั้งหมด client-side

WI-B04 — อนุมัติเบิกเงิน (Payroll Approval)

Boss ตรวจสอบคำขอเบิกเงินทั้งหมด และกดอนุมัติหรือปฏิเสธ

ข้อมูลที่เห็นต่อคำขอ
  • Pay ID + ชื่อช่าง + รอบ (15/30)
  • Gross Amount (ก่อนหักภาษี)
  • Tax 3% (จำนวนภาษี)
  • Net Amount (ยอดโอนจริง)
  • วันที่ยื่นคำขอ
การดำเนินการ
  • กดปุ่ม "อนุมัติ" → status = approved
  • ไม่มีปุ่มปฏิเสธ (Boss ต้องติดต่อช่างเอง)
  • ประวัติทั้งหมดดูได้ใน PAYROLL sheet
Backend: POST action=approve_payment, pay_id="PAY-XXX" → update col8 ของ PAYROLL เป็น "approved"

WI-B05 — รายงาน P&L รายเดือน (Profit & Loss)

วิเคราะห์กำไร-ขาดทุนจริงของร้านในแต่ละเดือน

สูตรคำนวณ
  • Revenue = Σ budget (งาน finished)
  • Material Cost = Σ material_cost
  • Labor Cost (ช่าง 40%) = Σ budget × 0.4
  • Net Profit = Revenue − Material Cost − Labor Cost
วิธีดูรายงาน
  • กดแท็บ "กำไร-ขาดทุน"
  • ระบบจัดกลุ่มตามเดือน (YYYY-MM)
  • ดูกราฟเปรียบเทียบรายเดือนในแท็บ "ภาพรวม"
Frontend: คำนวณ client-side จาก jobs array → group by month ของ created_at

WI-B06 — จัดการสิทธิ์ผู้ใช้ (User & Role Management)

เพิ่มช่างใหม่ เปลี่ยน role จาก customer → tailor หรือ boss

วิธีเพิ่มช่างใหม่
  • ให้ช่างสมัครสมาชิกผ่าน index.html ก่อน
  • Boss ไปที่แท็บ "จัดการสิทธิ์"
  • ค้นหาชื่อผู้ใช้งาน
  • กด Dropdown เลือก role ใหม่
  • บันทึก Log อัตโนมัติ
Role ที่มีในระบบ
  • customer — สั่งงาน / ติดตามสถานะ
  • tailor — รับงาน / อัปเดต / เบิกเงิน
  • boss — เข้าถึงทุกระบบ
Backend: POST action=update_role, user_id, role → update col4 USERS → logEvent(UPDATE_ROLE)

WI-B07 — วิเคราะห์งานปฏิเสธ (Rejection Analysis)

Boss ดูรายการงานที่ถูกปฏิเสธพร้อมเหตุผล เพื่อปรับปรุงกระบวนการ

วิธีใช้
  • ไปที่แท็บ "วิเคราะห์ปัญหา"
  • ดูรายการงาน status=rejected
  • ดูเหตุผลแต่ละงาน
  • ดู dispute_notes จากลูกค้า
ประโยชน์
  • ระบุปัญหาสต๊อกที่หมดบ่อย
  • ดูช่างที่ปฏิเสธงานบ่อยผิดปกติ
  • ปรับราคาหรือประเภทงานที่รับ
Frontend: filter jobs ที่ status=rejected หรือ cancelled → แสดง cancellation_reason

WI-B08 — เพิ่มค่าใช้จ่ายงาน (Additional Cost)

Boss ปรับราคาสุดท้ายของงาน กรณีมีค่าใช้จ่ายเพิ่มเติมที่ไม่ได้ตกลงในตอนแรก

การใช้งาน
  • เปิดใบงานใดก็ได้
  • กรอกจำนวนเงินที่เพิ่มเติม
  • กด "อัปเดตราคา"
ผลลัพธ์
  • budget ใหม่ = budget เดิม + additional_cost
  • กระทบ P&L และค่าแรงช่าง 40%
  • Log: UPDATE_COST บันทึกไว้
Backend: POST action=update_additional_cost, id, additional_cost → final = budget + additional → update col8 JOBS

WI-S05 — ระบบ Activity Log

ทุก action สำคัญถูกบันทึกลง LOGS sheet ใน Google Sheets อัตโนมัติ

Format แต่ละ Row ใน LOGS
  • timestamp — วันเวลา (Date object)
  • user_id — UUID ของผู้ทำ
  • username — ชื่อผู้ใช้
  • action — รหัสกิจกรรม
  • details — รายละเอียด
  • status — success / error
Action Codes ที่บันทึก
  • UPDATE_STATUS — เปลี่ยนสถานะงาน
  • RESET_PASSWORD — กู้คืนรหัสผ่าน
  • CHANGE_PASSWORD — เปลี่ยนรหัสผ่าน
  • UPDATE_ROLE — เปลี่ยน role
  • PAY_FINAL — ลูกค้าชำระเงิน
  • UPDATE_COST — Boss เพิ่มค่าใช้จ่าย
ฟังก์ชัน: logEvent(ss, userId, username, action, details, status) → appendRow ลง LOGS sheet

WI-S06 — ข้อควรทราบด้านความปลอดภัย (Security Notes)

สิ่งที่ระบบปกป้อง
  • Password ไม่ถูกส่งกลับใน API Response (getSheetData ข้ามคอลัมน์ password)
  • Reset password ต้องใช้ phone ยืนยัน
  • Change password ต้องรู้ password เดิม
  • Email แจ้งทุกครั้งที่ Boss login
ข้อจำกัดที่ต้องระวัง
  • Password เก็บเป็น plain text ใน Sheets
  • Demo admin/1234 เป็น hardcode ใน script.js
  • ไม่มี rate limiting / brute force protection
  • Role check ทำฝั่ง frontend เท่านั้น (ไม่มี server-side auth)
  • URL ของ Web App เป็น public สามารถ POST ได้โดยไม่ต้อง login
Production Advice: ระบบนี้เหมาะสำหรับร้านขนาดเล็ก/ทีมภายใน — ถ้า scale ขึ้นควรเพิ่ม server-side auth token และเข้ารหัส password (bcrypt)

WI-S07 — ข้อจำกัดของ Google Apps Script

ขีดจำกัดประสิทธิภาพ
  • Execution time สูงสุด 6 นาที/request
  • ข้อมูลเยอะ: getDataRange() ช้า
  • ไม่มี connection pooling
  • 1 request ต่อ 1 user ต่อเวลา
Real-time
  • ไม่มี WebSocket / Push
  • ต้อง refresh หน้าหรือ poll
  • Frontend โหลดข้อมูลทั้งหมด 1 ครั้งตอนเปิด
  • หมุนวนทุก X วินาทีได้ถ้าต้องการ
Quota (Free Tier)
  • Email: 100 ฉบับ/วัน
  • Url Fetch: 20,000 ครั้ง/วัน
  • Execution: 6 นาที/ครั้ง
  • Script Runtime: 90 นาที/วัน

Quick Reference — สรุปด่วน

ลูกค้า (Customer) ทำอะไรได้บ้าง
  • สมัครสมาชิก / Login / ลืมรหัสผ่าน
  • สั่งงานจ้าง + กรอกสัดส่วนร่างกาย
  • ดูสถานะงานของตัวเอง
  • ดูรูปความคืบหน้า
  • ยกเลิกงาน (pending/accepted เท่านั้น)
  • ยืนยันรับงาน + ให้ Rating 1-5 ดาว
  • ส่งเคลม/ข้อร้องเรียน
ช่าง (Tailor) ทำอะไรได้บ้าง
  • Login ผ่าน tailor.html เท่านั้น
  • ดูงานใหม่ + รับงาน + เลือกสต๊อก
  • ปฏิเสธงาน + ระบุเหตุผล
  • อัปเดตสถานะ (sewing) + อัปโหลดรูป
  • แจ้งงานเสร็จ (finished)
  • ดูรายได้ตัวเอง
  • เบิกเงินรอบ 15/30
Boss ทำได้ทุกอย่างของช่าง บวกเพิ่ม
  • Dashboard สรุปวันนี้ + สถิติรวม + กราฟ
  • จัดการสต๊อก (เพิ่ม/แก้ไข/กรอง)
  • ดูผลงานช่างทุกคน + ค่าแรง
  • อนุมัติเบิกเงิน + ดูภาษีหัก 3%
  • รายงาน P&L รายเดือน
  • จัดการ Role ผู้ใช้ทั้งหมด
  • วิเคราะห์งานปฏิเสธ + เพิ่มค่าใช้จ่าย
ระบบทำอะไรอัตโนมัติ
  • สร้าง Sheet 7 แผ่นถ้ายังไม่มี
  • ออก RN-XXXX-XXXX ทุกใบงานใหม่
  • ส่ง Email ทุก Event สำคัญ
  • หักสต๊อกและคำนวณต้นทุนวัสดุ
  • เตือน Email เมื่อสต๊อกใกล้หมด
  • คำนวณภาษีหัก 3% ทุกการเบิกเงิน
  • บันทึก Log ทุก action สำคัญ

pending หรือ accepted cancelled → ลูกค้ายกเลิก + เหตุผล
